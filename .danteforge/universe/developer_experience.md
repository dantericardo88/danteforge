## OSS Leader
**Name**: Pydantic-AI
**URL**: https://github.com/pydantic/pydantic-ai
**Key capability**: Production-grade, provider-agnostic agent framework with typed dependencies, typed outputs, YAML/JSON agent specs, composable capabilities, OpenTelemetry observability, evals, durable execution, MCP/A2A integration, and graph-backed execution. For DanteForge, the transferable capability is not code generation; it is the developer platform layer that makes any agent workflow typed, inspectable, replayable, overridable, and testable.
**Implementation approach**: Pydantic-AI puts DX in concrete APIs rather than prompt convention. `pydantic_ai_slim/pydantic_ai/agent/__init__.py` exposes `Agent(deps_type=..., output_type=..., tools=..., toolsets=..., capabilities=..., metadata=..., retries=...)` so IDEs, type checkers, and AI coding assistants see the contract. `Agent.from_spec()` and `Agent.from_file()` load YAML/JSON specs; `Agent.instrument_all()` enables tracing globally; `Agent.iter()` returns a graph-backed `AgentRun`; `AgentRun.next()` lets a caller inspect, mutate, or skip a node before continuing; `Agent.override()` provides scoped model/tool/dependency substitution for tests; `Agent.to_cli()` turns an agent into a terminal harness. `pydantic_graph/pydantic_graph/graph.py` supplies the finite-state-machine layer with `Graph.iter()`, `GraphRun.next()`, persistence hooks, and `Graph.mermaid_code()` for visual inspection. Secondary OSS leader: LangGraph, whose `StateGraph` builder, checkpointing, interrupts, durable execution, human-in-the-loop state editing, and time-travel debugging are the best public orchestration patterns for long-running coding workflows.

## What practitioners say (Reddit/Twitter/HN)
- Reddit complaints cluster around state visibility and deterministic activation. r/LocalLLaMA users analyzing 300+ agent failures named missing agent-state observability, silent tool failures, fragmented multi-agent traces, and hallucinated execution paths with no audit trail. r/ClaudeAI users like skills and bundled scripts, but report that skills are "hit or miss" unless explicit hook/home-instruction activation patterns force the model to notice them.
- HN and X commentary converges on typed control plus operational surface area. HN developers praise Pydantic-AI for easy agent construction, validation, dependency injection, and a better universal interface than lower-level LLM wrappers, while warning that lower-level APIs can still be painful. X power users frame 9+ Claude Code DX as hooks, worktrees, custom agents, remote dispatch, browser checks, session forking, batch runs, and kanban/worktree orchestration that makes parallel agents manageable instead of invisible terminal chaos.

## Key techniques that separate 9+ from 7
1. Typed contracts with scoped overrides. Copy Pydantic-AI's `Agent.from_spec()`, `Agent.from_file()`, typed `deps_type`/`output_type`, and `Agent.override()` patterns into DanteForge: every workflow packet should validate against a schema, every adapter dependency should be injectable, and every provider/tool/browser/GitHub integration should have a fake override usable in CI.
2. Unified node-level trace and replay. Copy Pydantic-AI `Agent.iter()`/`AgentRun.next()` and LangGraph checkpoint/time-travel patterns: persist one ordered trace with nodes such as `spec_loaded`, `skill_candidates`, `skill_selected`, `adapter_selected`, `agent_dispatched`, `tool_call`, `file_patch`, `test_run`, `judge_verdict`, `merge_decision`, and `resume_ready`, each with span IDs, timestamps, prompt hashes, redacted inputs, outputs, file diffs, and capability-test receipts.
3. Deterministic skill/hook orchestration. Practitioner reports show that "skills exist" is not enough. DanteForge needs a skill activation planner that logs `candidateSkillIds`, `selectedSkillIds`, `activationReason`, `rejectedReason`, and `forcedByUser`, plus lifecycle hooks equivalent to `before_node_run`, `wrap_node_run`, `after_node_run`, and `on_node_run_error` around command start, model call, tool call, edit, test, judge, and final verdict.

## Builder checklist for 9+
- [ ] Add a typed run schema for every workflow command with `runId`, `dimensionId`, `workPacketId`, `assistantProvider`, `adapter`, `model`, `specHash`, `planHash`, `skillIds`, `hookIds`, `leaseId`, `worktreePath`, `startedAt`, `completedAt`, `status`, `filesTouched`, `commandsRun`, `capabilityTests`, `cost`, `tracePath`, and `resumeCommand`.
- [ ] Emit `.danteforge/runs/<runId>/trace.jsonl` and `.danteforge/runs/<runId>/README.md`; include stable event names, parent span IDs, selected skills, rejected skills, tool inputs/outputs with redaction, command exit codes, evidence paths, and exact retry/resume commands.
- [ ] Implement scoped adapter/tool overrides like `Agent.override()` so tests can run fake LLMs, fake GitHub, fake browser, fake filesystem claims, and fake matrix leases without network access or real provider billing.
- [ ] Add deterministic skill activation receipts before dispatch: candidate skills, match rules, chosen skills, rejected skills, token estimate, whether activation was explicit or inferred, and the command/hook that enforced it.
- [ ] Add graph inspection for multi-agent waves: Mermaid output plus machine-readable DAG nodes for spec, plan, task, worker lease, tool call, tests, judge, locked merge, and Time Machine receipt.
- [ ] Add DX smoke tests that run a fake wave, inject one node failure, resume it, verify the trace remains continuous, and assert that the same workflow spec runs against at least two adapters.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can run one fake-provider command, open one trace directory, see every selected skill/hook/tool/test/evidence event in order, resume after an injected failure, and prove the same workflow spec works across at least two assistant adapters without editing provider-specific workflow logic.
**Red flags capping at <=7**: Important events only exist in console logs; skills are selected by hidden prompt behavior with no receipt; failed runs lack resume commands; traces omit tool failures, command exit codes, or file diffs; provider-specific assumptions leak into generic workflow code; tests require live LLM calls; docs describe workflows that CI cannot execute.
**Evidence to look for in a diff**: New run/spec trace schema; tests for schema validation and fake adapter overrides; JSONL trace writer with stable event names; CLI output that prints trace path and resume command; skill activation receipts; Mermaid/DAG generation; redaction tests for prompts/tool output; and an end-to-end fake-provider test covering fail, inspect, resume, and pass.
