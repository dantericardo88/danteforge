## OSS Leader
**Name**: Aider, with BMad Method as the closest meta-layer peer
**URL**: https://github.com/Aider-AI/aider
**Key capability**: Aider converts adoption into measurable trust loops: visible GitHub/PyPI/OpenRouter badges, a Discord community, release notes, blog/tutorial links, a public LLM leaderboard, opt-in analytics, testimonials from recognizable developers, and a benchmark harness that lets users compare models on coding outcomes instead of marketing claims. BMad Method is the stronger meta-layer pattern: cross-assistant installation, modules, web bundles for ChatGPT/Gemini, Discord/YouTube/Discussions, and a marketplace-like module ecosystem.
**Implementation approach**: Aider's adoption engine is implemented as product telemetry plus public proof. `aider/analytics.py` uses `compute_hex_threshold()` and `is_uuid_in_percentage()` to sample a stable 10 percent UUID cohort, `Analytics.need_to_ask()` to avoid nagging users, `_redact_model_name()` to avoid leaking unknown model names, and `Analytics.event()` to send typed PostHog events or append local JSONL telemetry when `logfile` is set. `benchmark/benchmark.py` turns model quality into shareable evidence: `run_test_real()` copies polyglot benchmark tasks, drives `Coder.create()`, runs unit tests through `run_unit_tests()`, writes `.aider.results.json`, and `summarize_results()` emits pass rate, malformed response count, cost, duration, token counts, and command metadata for leaderboard rows. `aider/coders/base_coder.py` reinforces adoption with trust ergonomics: `Coder.get_announcements()` prints model/repo-map/git context at startup, `show_usage_report()` emits cost/token analytics, `auto_commit()`/`show_undo_hint()` make AI changes auditable, and `allowed_to_edit()` asks before touching files outside the chat. BMad Method adds the meta-layer playbook DanteForge should copy: `tools/installer/commands/install.js` supports non-interactive `npx bmad-method install --directory ... --modules ... --tools ... --yes`, `IdeManager.setupBatch()` deduplicates shared target directories across Claude Code/Cursor/Codex/Windsurf/OpenCode/Copilot-style platforms, and `PluginResolver.resolve()` tries five marketplace resolution strategies before synthesizing module metadata from `SKILL.md` frontmatter.

## What practitioners say (Reddit/Twitter/HN)
- Reddit praise for Aider centers on control, speed, repo-map context, model choice, and fast rollback; complaints center on needing hands-on skill, missing MCP/agentic breadth, API-cost anxiety, and manual context selection. For DanteForge this means a 9+ adoption story must offer "Aider precision" plus easy-mode onboarding: assistant presets, workflow recipes, rollback receipts, and clear cost boundaries.
- Reddit/HN complaints about AI coding tools repeatedly mention codebase breakage, tool-call limits, agents touching too much, hidden cost, slow responses, and weak rollback. Positive comments praise git commits, restricted file access, inline diffs, tight context, and being able to use the tool inside existing IDE/editor workflows.
- X/HN power-user commentary says the adoption frontier is not another chat UI; it is hooks, worktrees, batch execution, remote dispatch, custom agents, skill folders, browser verification, and task-board visibility. Community adoption rises when advanced workflows are packaged as sharable setup directories and installer targets.
- HN commentary on Aider highlights the repo map, automatic edit application, git commits, and benchmark-driven engineering as reasons developers recommend it. HN commentary on coding assistants getting worse shows a split: some users want every AI change as a discrete commit, while others need controls to prevent unreviewed commits.

## Key techniques that separate 9+ from 7
1. Instrumented adoption funnel, not vanity metrics. Copy Aider's `Analytics.need_to_ask()` sampling and `_redact_model_name()` privacy pattern, but emit DanteForge events for `install_started`, `assistant_detected`, `skill_exported`, `workflow_started`, `gate_failed`, `gate_recovered`, `first_success`, `resume_used`, `community_link_opened`, and `support_bundle_created`. Store local `.danteforge/adoption/events.jsonl` even when remote telemetry is disabled.
2. Public proof loop. Copy Aider's leaderboard mechanics: every community claim should link to a reproducible benchmark, showcase, or receipt. Add `danteforge adoption report` that aggregates install target, assistant, workflow, duration, gates passed, tests run, failures recovered, cost estimate, and anonymized capability score into a shareable markdown card.
3. Cross-assistant distribution as the community primitive. Copy BMad's `install --modules --tools --yes`, `IdeManager.setupBatch()`, and `PluginResolver.resolve()` patterns so DanteForge can install the same workflow into Claude Code, Codex, Grok Build, Cursor, Aider, Goose, QwenCoder, and OpenCode without users hand-copying prompts.
4. Contributor conversion pipeline. A 9+ community needs issue labels, good-first-work packets, generated reproduction bundles, Discord-to-issue escalation, monthly community calls, and a module/skill submission validator. The diff should show code paths that create contributor-ready artifacts, not only README links.

## Builder checklist for 9+
- [ ] Add `src/core/adoption-telemetry.ts` with stable UUID cohort sampling, opt-in/disable state, model/provider redaction, local JSONL fallback, and event names for install, assistant export, workflow start, gate failure, recovery, first success, resume, and support bundle generation.
- [ ] Add `danteforge adoption report` to produce `.danteforge/adoption/REPORT.md` with installs by assistant, exported skills, workflow successes/failures, top failed gates, average time to first success, resume usage, support links clicked, and anonymized model/provider mix.
- [ ] Add a public showcase/benchmark receipt format under `.danteforge/adoption/showcases/<id>.md` that records assistant, repo size, workflow command, gates passed, tests run, diff summary, cost estimate, and reproducible command.
- [ ] Add cross-assistant installer receipts: after `setup assistants`, write one file listing target tools, copied skills/commands/prompts, skipped tools, conflicts, and exact paths so community support can diagnose broken installs from one paste.
- [ ] Add GitHub issue templates for bug, workflow recipe, assistant integration, skill/module submission, and showcase; each template should request `danteforge doctor`, adoption report path, assistant target, OS, Node version, and reproduction command.
- [ ] Add community-health automation: stale labels, first-time contributor greeting, weekly issue triage summary, release-note generator that includes new assistants/modules/skills, and a docs page showing where to ask Discord vs GitHub Discussions vs issues.
- [ ] Add a skill/module registry validator modeled on BMad's fallback resolver: accept root module files, setup-skill assets, standalone skills, or synthesized metadata from `SKILL.md` frontmatter; reject packages without name, description, activation rules, license, and smoke test.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can install DanteForge into at least three assistant targets, run one workflow, opt into or out of telemetry, open one adoption report, verify local JSONL events for install/start/gate/success/resume, see a generated setup receipt with exact copied files, publish a showcase receipt, and submit a validated third-party skill/module without maintainer handholding.
**Red flags capping at <=7**: Adoption evidence is only GitHub stars, README claims, or Discord links; no local telemetry exists; telemetry is remote-only or not privacy-reviewed; support cannot diagnose installs from a receipt; assistant integrations require manual prompt copying; no public benchmark/showcase format exists; community submissions bypass validation; issue templates do not collect reproduction data.
**Evidence to look for in a diff**: New adoption telemetry module with tests for cohort sampling, opt-out, redaction, and JSONL fallback; CLI command that writes `.danteforge/adoption/REPORT.md`; assistant setup receipt writer; issue templates; registry validator for skills/modules; docs linking community channels to exact use cases; benchmark/showcase receipt schema; CI tests proving telemetry works without network access.
