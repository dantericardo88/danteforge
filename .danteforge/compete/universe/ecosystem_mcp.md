## OSS Leader
**Name**: Agno
**URL**: https://github.com/agno-agi/agno
**Key capability**: Production agent-platform layer with provider-agnostic agents, teams, workflows, 100+ toolkits, MCP client integration, context providers, human approval, tracing, RBAC, multi-tenant isolation, and deployable AgentOS surfaces. For DanteForge, the transferable capability is not "an agent that codes"; it is the ecosystem adapter layer that lets any assistant or agent platform consume third-party tools, live context, prompts, and docs through stable MCP/tool abstractions while retaining run state, permissions, and observability.
**Implementation approach**: Agno's MCP integration is concrete and reusable. `libs/agno/agno/tools/mcp/mcp.py` defines `MCPTools`, a `Toolkit` subclass that supports `stdio`, `sse`, and `streamable-http`; validates transport/server parameter combinations; merges `get_default_environment()` with caller env for subprocess MCP servers; accepts `include_tools` and `exclude_tools` filters; supports `tool_name_prefix`; and exposes `refresh_connection` for reloading a server and its tool list on each run. `MCPTools.connect()` calls `_connect()`, establishes the transport with `stdio_client`, `sse_client`, or `streamablehttp_client`, creates a `ClientSession`, then `initialize()` calls `build_tools()`. `build_tools()` calls `session.list_tools()`, applies filters, wraps each remote MCP tool with `get_entrypoint_for_tool()`, and registers an Agno `Function` with the remote tool's `name`, `description`, `inputSchema`, HITL flags such as `requires_confirmation`, and control flags such as `stop_after_tool_call` and `show_result`. The strongest pattern is dynamic identity: `_call_header_provider()` inspects a provider function signature and passes `run_context`, `agent`, or `team`; `get_session_for_run()` creates a per-run MCP session keyed by `run_context.run_id`, merges dynamic headers into SSE/streamable HTTP connection config, serializes session creation with `_session_creation_lock`, and reclaims stale sessions with `_cleanup_stale_sessions()` plus a 300-second TTL. `libs/agno/agno/tools/mcp/multi_mcp.py` generalizes this across multiple servers with `AsyncExitStack`, `(run_id, server_idx)` session keys, `allow_partial_failure`, and per-server transport lists, though it is deprecated in favor of multiple `MCPTools` instances. Secondary OSS pattern to copy: Cloudflare Agents SDK Code Mode, which collapses huge MCP/API surfaces to `search()` and `execute()` tools backed by a typed OpenAPI view and a sandboxed Worker isolate, keeping token cost fixed instead of injecting thousands of tool schemas.

## What practitioners say (Reddit/Twitter/HN)
- Reddit practitioners split MCP use into two camps. r/ClaudeCode users complain that many MCP setups are worse than plain CLIs because they add auth pain, context noise, abstraction, and debugging friction; the actionable pattern is "MCP for access, skills/commands for use." The most useful comment thread calls out `defer_loading`, tool search, and innate server-side search as the difference between useful MCP and context pollution. r/ClaudeAI builders praise local codebase-context MCP servers when they compute something a skill cannot: semantic repo indexing, "best examples to follow", patterns to avoid, affected-file preflight, and low-confidence warnings before edit.
- HN commentary is more operational. MCP server authors complain that the spec moves fast, implementation docs lag, OAuth scopes can be undocumented, and remote MCP previously required `mcp-remote` proxies. Other HN users ask which MCP servers are actually useful for coding instead of Jira/admin work; practical answers mention Context7, fetch/doc tools, and LLM-consultation MCPs for second opinions. The repeated failure mode is loading large tool surfaces and then trusting the model to pick correctly without search, policy, or budget.
- X/Twitter developer commentary converges on token economy. Apify's MCPC launch frames current MCP pain as context overload, auth pain, and failed tool calls, then proposes a universal CLI client with OAuth, sessions, JSON mode, and proxy support. Obie Fernandez's Code Mode thread describes a 42-tool MCP server burning roughly 18k tokens before useful work and points to the Cloudflare pattern: expose two tools, `search` and `execute`, instead of dozens or thousands. Sebastian Raschka's Claude Code analysis highlights context-bloat mitigation, file-read deduplication, writing large tool results to disk with previews, structured session memory, and forked subagents as part of why ecosystem tooling feels better than plain chat.

## Key techniques that separate 9+ from 7
1. Lazy, searchable tool discovery instead of eager schema injection. Copy Cloudflare Code Mode and OpenAI hosted MCP `defer_loading` patterns: large MCP/API surfaces should expose `search_tools(query, constraints)` and `execute_tool(toolRef, args)` or `search()` and `execute()` rather than dumping every JSON schema into the system prompt. DanteForge should compute `schemaTokenEstimate`, cap eager tools, and force search-first mode above a threshold such as 25 tools or 8k schema tokens.
2. Per-run identity, policy, and session lifecycle. Copy Agno `header_provider`, `get_session_for_run()`, TTL cleanup, include/exclude filters, and `tool_name_prefix`. Each DanteForge run should create an integration session keyed by `runId`, `assistantProvider`, `adapter`, and `workPacketId`; inject short-lived OAuth/API headers outside the model context; apply role-based tool allowlists; log duplicate names; and clean stale sessions deterministically.
3. Integration evidence and recovery semantics. Copy the production MCP paper's CABP/ATBA/SERF ideas: identity-scoped broker routing, adaptive timeout budgets for sequential tool chains, and machine-readable error classes. DanteForge should record `mcp.list_tools`, `mcp.call_tool`, `mcp.auth_refresh`, `mcp.retry`, `mcp.approval_required`, `mcp.fallback_cli`, and `mcp.error` trace events with latency, token cost, auth principal, policy decision, retry count, and redacted request/response previews.
4. Security by default for tool metadata and execution. Copy MCP threat-model guidance: statically scan tool names/descriptions/schemas for prompt-injection instructions, hidden exfiltration instructions, overbroad filesystem/network scopes, and ambiguous destructive verbs. Generated integration code must run in a sandbox with no default filesystem, no ambient env vars, and explicit outbound network policy, matching Cloudflare's Worker isolate pattern.

- [ ] Add per-run dynamic auth sessions keyed by `runId` with TTL cleanup, role-based allowlists, redacted headers, and trace events for auth creation, refresh, failure, and cleanup.
- [ ] Add Code Mode wrapper support for OpenAPI-backed integrations: `search(code)` over a resolved spec, `execute(code)` against an authenticated API client, and sandbox controls for filesystem, environment, network, timeout, and output size.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can run a non-live fake-provider integration test plus one real local MCP smoke test and see receipts proving stdio and streamable HTTP support, per-run dynamic auth or a redacted fake equivalent, include/exclude filters, duplicate tool detection, lazy/search-first discovery for a large tool surface, sandboxed Code Mode execution, structured error categories, and identical integration manifests consumed by at least three assistant adapters.

**Evidence to look for in a diff**: New integration manifest schema; MCP preflight command; `list_tools`/`call_tool` trace events; schema token estimator; search-first router; Code Mode `search`/`execute` wrapper; dynamic auth/session manager with TTL cleanup; policy allowlist/blocklist tests; tool metadata security scanner; fake MCP server fixtures; adapter parity tests; and a generated receipt under `.danteforge/integrations/` showing tool counts, token estimates, transports, auth state, policy decisions, and errors.
