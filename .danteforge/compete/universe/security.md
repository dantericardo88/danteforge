## OSS Leader
**Name**: OpenHands (All-Hands AI), with OpenSandbox as the strongest secondary runtime-security reference
**URL**: https://github.com/OpenHands/OpenHands
**Key capability**: OpenHands hardens AI coding by putting untrusted agent work behind layered controls: sandboxed runtimes, per-sandbox session API keys, LLM action-risk analysis, user confirmation for risky actions, secret lookup scoped to a running sandbox, and visible security-analyzer logs. OpenSandbox adds the best OSS pattern for portable runtime isolation: Docker/Kubernetes lifecycle control, secure runtimes such as gVisor/Kata/Firecracker, deny-by-default egress policy, nftables enforcement, and sidecar-based network control.
**Implementation approach**: OpenHands' security path is not a single scanner. It combines policy, runtime boundary, and UX confirmation. The OpenHands SDK security docs expose `openhands.sdk.security` patterns such as `SecurityRisk`, `LLMSecurityAnalyzer`, deterministic invariants, and `ConfirmRisky`; the implementation lesson is to classify action risk before execution and map the result through a confirmation policy. In the public OpenHands repo, `openhands/app_server/sandbox/session_auth.py` rejects missing or invalid `X-Session-API-Key`, rejects keys for non-running sandboxes in `validate_session_key`, and enforces owner match in `validate_session_key_ownership`; `openhands/app_server/sandbox/docker_sandbox_service.py` creates 32-byte random per-sandbox session keys, injects them as `OH_SESSION_API_KEYS_0`, maps ports, and waits for `/alive` before declaring the agent server ready; `frontend/src/stores/security-analyzer-store.ts` persists analyzer events with `security_risk`, `confirmation_state`, and `confirmed_changed` so the operator can see whether a risky command was awaited, confirmed, or rejected. OpenSandbox's transferable patterns are `components/egress/pkg/policy/policy.go` (`DefaultDenyPolicy`, `ParsePolicy`, `NetworkPolicy.Evaluate`, `StaticIPSets`) and `components/egress/pkg/nftables/manager.go` (`Manager.ApplyStatic`, dynamic resolved-IP sets, DoT/DoH blocking, output-chain default drop/accept policy).

## What practitioners say (Reddit/Twitter/HN)
- Reddit practitioners repeatedly focus on ambient authority. r/LocalLLaMA discussions about running agent tools locally praise containerization but ask for stronger network and filesystem isolation; r/ClaudeAI and r/ChatGPT users complain that coding agents can run broad shell commands or touch real projects unless permissions and worktrees are configured carefully. The actionable pattern is not "trust the model"; it is deny by default, isolate the workspace, and make dangerous tool use visible before execution.
- HN and X commentary around AI coding agents is skeptical of hidden autonomy. Developers praise tools that show diffs, ask before destructive commands, and run in sandboxes, while warning that agents with shell, browser, GitHub, and network access create prompt-injection and confused-deputy risk. For DanteForge, this means security hardening must be orchestration-level: every provider adapter must pass through the same risk classifier, policy gate, sandbox allocator, secret broker, and audit writer.

## Key techniques that separate 9+ from 7
1. **Action-risk classification before execution**. Copy OpenHands' pattern: represent risk as a typed enum (`UNKNOWN`, `LOW`, `MEDIUM`, `HIGH`), run deterministic invariants for obvious dangerous operations, optionally ask an LLM analyzer for ambiguous shell/code actions, then map the result through a confirmation policy like `ConfirmRisky`. DanteForge should log `riskSource=deterministic|llm|fallback`, `riskReason`, `policyDecision`, and `confirmationState` for every shell, file write, git, browser, GitHub, MCP, and network tool call.
2. **Sandbox/session boundary instead of ambient process trust**. OpenHands uses random per-sandbox session keys and rejects keys when the sandbox is not `RUNNING`; OpenSandbox adds runtime-level isolation, resource limits, secure runtimes, and sidecar network policy. DanteForge should allocate a `sandboxId`/`sessionKeyHash` per work packet, scope secret lookup to that sandbox, deny access after pause/delete, and run capability tests that prove leaked or stale session keys cannot fetch secrets.
3. **Deny-by-default network and secret egress controls**. OpenSandbox's `DefaultDenyPolicy`, domain/IP rule evaluation, static IP sets, nftables output-chain enforcement, dynamic resolved-IP allow sets, and DoT/DoH blocking are the concrete bar. DanteForge 9+ needs per-work-packet egress policy receipts, explicit allowed domains for GitHub/npm/docs, redacted command logs, secret-use receipts with name-only disclosure, and a failing test for attempted exfiltration to an unapproved host.

## Builder checklist for 9+
- [ ] Add a central security gate around every adapter tool call with fields `runId`, `leaseId`, `assistant`, `tool`, `actionKind`, `argsHash`, `risk`, `riskSource`, `riskReason`, `policyDecision`, `confirmationState`, `sandboxId`, and `receiptPath`.
- [ ] Implement deterministic risky-action rules for `rm -rf`, shell redirection outside the worktree, `curl|bash`, credential file reads, `git push`, package publish, network exfiltration, broad recursive writes, and edits to protected paths; route ambiguous actions to an optional LLM analyzer.
- [ ] Add policy classes equivalent to `AlwaysConfirm`, `NeverConfirm`, and `ConfirmRisky`, with tests proving `HIGH` and `UNKNOWN` actions cannot execute in auto mode without an explicit signed approval receipt.
- [ ] Scope secrets through a broker that returns values only to a running sandbox/session; logs must record secret names and hashes, never raw values, and stale session keys must fail with 401/403-style errors.
- [ ] Add deny-by-default egress policy per work packet with allowlist entries for only required hosts; record DNS/IP decisions and block DoT/DoH bypasses where supported.
- [ ] Add adversarial security capability tests: prompt-injection attempts to read secrets, write outside the claim path, touch `.danteforge/compete/matrix.json`, bypass confirmation, access a denied domain, and reuse a paused sandbox key.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can run one security capability test that dispatches a malicious work packet and proves all of the following from receipts: risky actions are classified before execution, high/unknown risk actions pause or fail closed, stale sandbox/session keys cannot retrieve secrets, denied egress is blocked, raw secrets are absent from logs, protected paths are untouched, and the same gate wraps at least two provider adapters.
**Red flags capping at ≤7**: Security is only prompt text; no central wrapper around tool execution; no typed risk enum; no confirmation receipt; logs contain raw command arguments or secrets; network is unrestricted; stale sandbox/session tokens remain valid; dangerous actions are only blocked in one assistant adapter; tests assert file existence rather than executing hostile workflows.
**Evidence to look for in a diff**: New `security-gate` or `policy` module; typed action-risk model; adapter/tool middleware; risk analyzer tests; confirmation-policy tests; sandbox/session-key lifecycle checks; secret redaction snapshots; egress allow/deny tests; malicious work-packet fixtures; JSONL audit receipts; capability_test command that runs the hostile scenario end to end and exits non-zero on any bypass.
