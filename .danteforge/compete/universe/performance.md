## OSS Leader
**Name**: OpenHands (All-Hands AI), with SWE-agent/DARS as the strongest secondary coding-agent performance reference
**URL**: https://github.com/OpenHands/OpenHands
**Key capability**: OpenHands is the best OSS reference for making AI software agents faster and cheaper as an orchestration platform: provider-agnostic SDK, local or remote workspaces, event-backed conversations, prompt-cache pinning, context condensers, metrics/cost tracking, parallel tool execution, sub-agent delegation, stuck detection, and forkable conversations. The transferable capability for DanteForge is not "write code better"; it is an optimizer layer that measures and controls agent throughput, token burn, latency, retries, context growth, and parallelism across assistants.
**Implementation approach**: OpenHands V1 moves performance controls into explicit runtime primitives. `openhands-sdk/openhands/sdk/conversation/impl/local_conversation.py` uses lazy plugin loading in `_ensure_plugins_loaded()`, one-time agent initialization in `_ensure_agent_ready()`, a bounded `Conversation.run()` loop with `max_iteration_per_run`, `StuckDetector.is_stuck()` checks, `CancellationToken`, stop hooks, and `ConversationErrorEvent` on max-iteration failure. The same file pins OpenAI prefix-cache locality with `_pin_prompt_cache_key()` by setting `agent.llm._prompt_cache_key = conversation_id`, and `fork()` deep-copies event history while optionally resetting metrics so alternative branches can reuse context without corrupting the source run. The SDK docs expose `tool_concurrency_limit` on `Agent`; persisted `ActionEvent.llm_response_id` batches are used to prove multiple tools ran from the same LLM response. `openhands/memory/condenser/impl/conversation_window_condenser.py` keeps the system message, first user message, recall action/observation, and recent non-dangling action-observation pairs while emitting a `CondensationAction` for forgotten event IDs. `openhands/memory/condenser/impl/llm_attention_condenser.py` asks an auxiliary LLM for `ImportantEventSelection.ids`, trims to half the configured history, disables prompt caching for the condenser LLM, and records condenser LLM metrics. Secondary reference: SWE-agent plus DARS/AgentLens show process-level performance optimization through trajectory logging, retry/review loops, dynamic action re-sampling, waste-signal detection, and pass@k-style inference-time scaling.

## What practitioners say (Reddit/Twitter/HN)
- Reddit practitioners are split: parallel agents are praised when each agent gets an isolated worktree, branch, port range, and independent task, but users warn that multi-agent fanout burns tokens and creates chaos when work is tightly coupled. r/ClaudeAI threads describe 2-5 Claude Code sessions in git worktrees as a real speedup, while another thread says subagents can cost about 15x tokens unless cloned agents share an identical cached prefix; commenters also call out port conflicts and context over 200k tokens as practical ceilings.
- HN and X commentary converges on measurable orchestration rather than raw model speed. HN complaints mention vision-based agents taking 2-10 seconds per action, token burn, hidden retry overhead, and poor runtime visibility; another HN cost breakdown says parsing/tool retries can push effective cost 40-60% above baseline. X commentary around Claude Code and Codex highlights lifecycle hooks, worktrees, batch/fork workflows, `--bare` startup speed, parallel tool calling, subagents, terse prompts, and native tool-calling as the performance alpha that meta-layer tools should adopt portably.

## Key techniques that separate 9+ from 7
1. Cache-aware orchestration with stable prefixes. DanteForge should copy OpenHands `_pin_prompt_cache_key()` and practitioner "clone inherits parent" economics: keep system prompts, tool schemas, skills, and model choice stable before fanout; log `cache_read_tokens`, `cache_write_tokens`, `promptCacheKey`, and `prefixHash`; and fail a performance gate when dynamic tool injection or mid-run memory edits break the shared prefix.
2. Bounded parallelism with isolation proofs. Copy OpenHands `tool_concurrency_limit` and persisted `llm_response_id` batch verification, but wrap it in DanteForge ownership rules: parallelize read-only research, file discovery, tests, and disjoint workstreams; require separate worktrees/ports for mutating agents; record write-claim overlap, merge conflicts, and rework. Sequentially run tightly dependent refactors and bug fixes.
3. Trajectory-level waste detection. Adopt DARS/AgentLens patterns: store action/observation trajectories, label steps as exploration, implementation, verification, or orchestration, and flag duplicate tool calls, repeated failed commands, no-op edits, late test execution, loops, and branches that consume budget without improving pass probability. Use this to trigger early exit, retry damping, or dynamic action re-sampling instead of blind max-wave execution.

## Builder checklist for 9+
- [ ] Add `.danteforge/perf/<runId>/trace.jsonl` with per-node `startedAt`, `endedAt`, `assistant`, `model`, `worktree`, `promptHash`, `prefixHash`, `cacheReadTokens`, `cacheWriteTokens`, `promptTokens`, `completionTokens`, `reasoningTokens`, `llmLatencyMs`, `toolLatencyMs`, `costUsd`, `retryCount`, `contextTokens`, `status`, and `evidencePath`.
- [ ] Implement a performance gate that runs a fixed fake-provider and live-provider benchmark twice: baseline sequential mode and optimized mode. It must report pass rate, p50/p90 wall time, total cost, token count, retries, duplicate tool calls, and cache hit ratio.
- [ ] Add cache-prefix receipts before any parallel wave: selected skills, tool schemas, model, provider, system prompt hash, project memory hash, and whether each child agent inherited the exact parent prefix.
- [ ] Add a `parallelismPlan` to work packets with `parallelizable=true|false`, dependency edges, owned write paths, required isolated ports, max concurrent agents, and reason for choosing single-agent vs fanout.
- [ ] Add stuck/retry damping receipts: repeated action fingerprint, repeated observation fingerprint, max retry budget, early-exit reason, cancellation reason, and whether the run switched model or strategy.
- [ ] Add trajectory waste labels and tests that inject duplicate file reads, repeated failing commands, and late verification; the judge should fail if waste is not detected in the trace.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can run one fixed benchmark command and see receipts proving the optimized DanteForge run preserved pass rate while improving at least one of wall-clock time or cost, with per-node token/cost/latency attribution, prompt-cache evidence, context-condensation evidence, bounded retries, stuck-loop handling, and isolated parallel work claims across at least two assistant adapters or one live adapter plus a fake adapter.
**Red flags capping at <=7**: Only total elapsed time is reported; "parallel" agents share one dirty worktree; no token/cost attribution per node; no cache-read/write metrics; no context condenser before/after counts; retries are unbounded; max-iteration failures are scored as performance wins; tool latency and LLM latency are mixed together; duplicate tool calls are invisible; tests require a live model with no fake-provider baseline; performance claims rely on screenshots or anecdotes.
**Evidence to look for in a diff**: New performance trace schema; benchmark fixture with deterministic fake adapter; per-node metrics writer; prompt-cache prefix hash; context condenser receipts; `parallelismPlan` validation; worktree/port isolation checks; retry/stuck detectors; trajectory waste classifier; CLI output with p50/p90/cost/pass-rate deltas; tests that fail when optimization lowers pass rate or hides retry/tool-call waste.
