## OSS Leader
**Name**: DanteForge self, with OpenAI Agents SDK guardrails and Railguard as reference patterns
**URL**: https://github.com/dantericardo88/danteforge
**Key capability**: Provider-agnostic governance for AI coding assistants: ordered workflow gates, scoring doctrine, capability-test score caps, harden/merge courts, completion-integrity checks, Time Machine provenance, and work-packet constraints that can sit above Claude Code, Codex, Grok Build, Cursor, Aider, Goose, or future coding agents. OpenAI Agents SDK is the best OSS guardrail API reference for tripwire semantics; Railguard is the best OSS runtime-reference for tool-call interception, path fencing, command classification, memory safety, and rollback around Claude Code.
**Implementation approach**: DanteForge already has concrete enforcement primitives. `src/core/gates.ts` defines `GateError`, `requireConstitution`, `requireSpec`, `requireClarify`, `requirePlan`, `requireTests`, `requireDesign`, `requireNoLoops`, `requireReflectionScore`, and `requireNoDrift`; these throw typed blocking errors with gate names, remedies, and error codes instead of relying on prompt reminders. `src/core/workflow-enforcer.ts` encodes `WORKFLOW_GRAPH` transitions such as `constitution -> specify`, `plan -> tasks`, and `tasks/design -> forge`, then `validateTransition()` and `enforceWorkflow()` run the required gate functions before a command may proceed. `src/core/scoring-doctrine.ts` centralizes evidence-only scoring rules and `wrapPromptWithDoctrine()` injects them into LLM prompts. `src/matrix/engines/capability-test-runner.ts` implements the Fix A hard cap with `runCapabilityTest()`, `applyScoreCap()`, and `runCapabilityTests()`: absent, malformed, `no_capability_test`, or failing capability tests cap scores above 5.0. `src/matrix/courts/merge-court.ts` layers policy gates in `runMergeCourt()`: LOC hard cap, zero-tolerance stub scan, security court, capability-test gate, conflict arbitration, and Time Machine event attachment before an approval is written. `src/core/completion-integrity.ts` exposes `runCIPCheck()`, which blocks frontier claims when capability tests fail, outcomes are absent or stale, stubs appear in critical paths, or cold outcome execution cannot support the stored score. The gap versus a 9+ leader is that these gates are strong but scattered; the next frontier is a single external policy engine around every agent action with append-only `policy_decision` events, pre-tool/pre-write enforcement, and deterministic replay.

## What practitioners say (Reddit/Twitter/HN)
- Reddit and HN practitioners consistently complain that permission fatigue pushes users toward unsafe "skip permissions" modes. The r/ClaudeAI thread on `--dangerously-skip-permissions` names the core failure mode: untrusted files, web pages, MCP outputs, or encoded payloads enter the agent context while the agent has trusted privileges. The actionable request is a middle ground: repo-scoped autonomy, hooks/rules for risky actions, sandboxing for untrusted IO, and logs that can be reviewed after a run.
- Reddit hook users praise small deterministic hooks over broad prompt instructions. Specific patterns mentioned include blocking edits before a file has been read, blocking commands outside the repo, preventing generated-file edits, rejecting merged-branch commits, running receipt scripts after edits, and using `PostToolUse`/`Stop` gates for formatter, typecheck, or test enforcement. A common warning is to keep hooks dumb and deterministic; if a hook needs judgment, move it into an explicit review step.
- HN commentary around Claude Code mishaps converges on external enforcement: locked-down sandboxes, no broad GitHub credentials in the agent environment, PreToolUse hooks for destructive commands, deterministic safeguards outside the LLM, and provenance for changes. Skeptics point out that prompt-level rules and probabilistic classifiers reduce risk but do not create a hard security boundary.
- X commentary around Claude Code hooks emphasizes the same operational primitives: `PreToolUse` as a security gate, `PostToolUse` for format/lint/validate, `Stop` as a "tests must pass before done" gate, and lifecycle hooks for notifications/logging. The strongest pattern for DanteForge is not copying Claude Code, but abstracting those hook semantics into a provider-neutral policy layer.

## Key techniques that separate 9+ from 7
1. Single policy engine instead of scattered gates. Add a `ConstitutionalPolicyEngine` that owns rule evaluation for `tool_call`, `file_write`, `score_write`, `merge_attempt`, `phase_transition`, and `completion_claim`. It should call existing primitives such as `requirePlan()`, `runCapabilityTest()`, `runCIPCheck()`, stub scan, security court, protected-line checks, and allowed-path checks through one typed interface.
2. Action-level receipts with replay. Every action should append a JSONL event like `{eventId, policyVersion, workPacketId, agentId, provider, actionType, actionSummary, ruleIds, decision, reason, inputHash, outputHash, timestamp}` before execution, followed by an execution result and verification receipt. A replay command should prove why an action was allowed or blocked without trusting the agent transcript.
3. External pre-action enforcement. A 9+ system gates before irreversible side effects. For Claude Code this maps to `PreToolUse`; for OpenAI Agents SDK it maps to blocking guardrails and tool guardrails; for DanteForge it should wrap shell/file/score/merge operations in the host adapter or MCP server, outside the prompt context. Prompt instructions remain explanatory, not load-bearing.

## Judge scoring criteria
**PASS at 9 requires**: A reviewer can run a cold governance capability test and see one session exercise at least three providers or adapter modes through one policy engine, with receipts proving pre-action blocking for an out-of-scope file write, destructive shell command, invalid score write, invalid merge, and invalid completion claim. The receipt chain must include `workPacketId`, `agentId`, `provider`, `policyVersion`, `ruleIds`, `decision`, `reason`, `executionResult`, and linked test or capability-test output.
**Red flags capping at <=7**: Governance appears only in prompt text or markdown instructions; `--light` or skip flags bypass gates without audit; score writes can happen by editing `matrix.json`; capability tests are structural existence checks only; merge/done decisions trust agent self-report; hooks only warn after the tool already ran; policy logs can be edited by the agent without detection; no test intentionally proves blocked actions.
**Evidence to look for in a diff**: New central policy engine types and rule ids; adapters that normalize provider tool calls into policy actions; append-only policy-event writer; pre-execution checks before shell/file/score/merge operations; tests that assert `block` decisions and exit codes; replay command; integration with `runCapabilityTest()`, `runCIPCheck()`, `runMergeCourt()`, protected-lines, and Time Machine causal commits.
